# Distributed Consensus Algorithms: Paxos, Raft, and Byzantine Fault Tolerance  

## 1. Introduction  

Distributed systems are built from many independent processes that cooperate to provide a service, but they are inevitably subject to component failures, network partitions, and, in some environments, arbitrary (Byzantine) faults. Achieving *consensus*—the task of getting all non‑faulty nodes to agree on a single value—is the cornerstone of many mission‑critical services such as replicated state machines, distributed databases, lock services, and blockchain ledgers.

The problem is formally defined by the **Consensus** abstraction:

> Given a set of proposals \(\{p_1,p_2,\dots\}\), every correct process must decide on exactly one proposed value, and all correct processes must decide on the same value.

In practice, we care about *safety* (never agree on two different values) and *liveness* (eventually make progress if a majority of nodes are available). The exact failure model determines which algorithm is appropriate. This essay examines three families of consensus protocols:

1. **Paxos** – the classic crash‑fault tolerant (CFT) algorithm, proven correct under the asynchronous model with a majority quorum.
2. **Raft** – a simplified, leader‑based CFT protocol designed for implementers; it is equivalent to Paxos but easier to reason about.
3. **Byzantine Fault Tolerant (BFT) protocols** – capable of handling arbitrary, malicious behavior, the most common instance being Practical Byzantine Fault Tolerance (PBFT).

Each section includes a high‑level description, core algorithm steps, and concrete message flow examples.

## 2. Background: Failure Models  

Before diving into algorithms we must be explicit about the *fault model*.

| Model | Assumptions on Nodes |
|-------|----------------------|
| **Crash‑fault (CFT)** | A node may stop at any time but never behaves incorrectly; it either works correctly or simply stops responding. |
| **Omission‑fault** | A node can fail to send or deliver messages, but otherwise follows the protocol. |
| **Byzantine fault** | A node may behave arbitrarily—omit messages, send incorrect data, collude with others, etc.—and we must still guarantee safety (provided a bound on the number of faulty nodes). |

Most production systems start with CFT because it yields lower latency; when malicious attackers or software bugs cannot be tolerated they augment the protocol to BFT.

## 3. Paxos  

### 3.1 The Original Algorithm  

Paxos, introduced by Leslie Lamport in 1998 (and refined in “Paxos Made Simple”), solves consensus under crash‑faults with a *quorum* of majority (or any majority of a known set). The protocol proceeds in two phases:

| Phase | Message | Purpose |
|-------|---------|---------|
| **Prepare** | `Prepare(n)` – sent by a proposer to all acceptors | The proposer asks for the highest-numbered proposal it has seen so far. Acceptors promise not to accept any proposal with a number lower than *n* and return any accepted value if one exists. |
| **Accept** | `Accept(n, v)` – from proposer to acceptors | After gathering a quorum of Prepare responses, the proposer may pick a value (usually the one with the highest round number) and ask acceptors to accept it. If an acceptor has already promised on *n*, it will accept *(n,v)*; otherwise it ignores it. |

If a majority of acceptors accept a proposal, the value is **chosen** – all correct nodes will eventually learn it.

#### 3.1.1 Example: Three‑Node Cluster  

Consider three acceptors \(A_1,A_2,A_3\). Node \(P\) (the proposer) wants to propose a new configuration `{ “logIndex”:5, “term”:4 }`.

```
P → A1, A2, A3 : Prepare(n=7)
```

*All* acceptors reply with `Promise(n=7, lastAccepted=null)` because they have not seen any previous proposal.

After receiving promises from a majority (say \(A_1\) and \(A_2\)), \(P\) now sends Accept:

```
P → A1, A2, A3 : Accept(n=7, v="{logIndex:5}")
```

\(A_1\) and \(A_2\) accept the proposal and send back `Accepted(7)`. Since two out of three (a majority) accepted, the value is now chosen. The remaining node \(A_3\) will eventually learn it via a subsequent learner or through a new leader that has learned the latest state.

### 3.2 Multi‑Paxos  

In practice, we need to agree on many values, not just one. **Multi‑Paxos** runs repeated instances of basic Paxos, using a *stable leader* (often called “the distinguished proposer”) which can propose successive log entries without re‑doing the Prepare phase each time. This yields a linearizable replicated log, similar to what ZooKeeper or etcd provides.

### 3.3 Guarantees  

- **Safety**: No two proposals with different values can be chosen in the same round; any value that becomes chosen is immutable.
- **Liveness**: Provided a majority of nodes are up and communication is eventually reliable, a value will eventually be chosen (the protocol does not guarantee termination in an asynchronous system – this aligns with the FLP impossibility result).

### 3.4 Limitations  

- Complex to implement correctly: subtle interactions between Prepare/Accept phases often lead to “Paxos implementation bugs”.
- The need for majority quorums prevents operation when less than half the cluster is reachable.

## 4. Raft  

Raft was designed by Diego Ongaro and John Ousterhout (2014) with three goals: *understandability*, *implementability*, and *correctness* under crash‑faults. It explicitly elects a **leader** that coordinates the log, simplifying the protocol dramatically.

### 4.1 Core Components  

Raft splits consensus into three sub‑problems:

| Sub‑problem | Role |
|-------------|------|
| **Leader Election** | Nodes start as followers; after a timeout they become candidates and request votes. The candidate with majority of votes becomes leader. |
| **Log Replication** | All client commands are sent to the leader, which appends them to its own log, then replicates them via `AppendEntries` messages to followers. |
| **Safety** | Ensures that logs are consistent: a value can only be committed if it has been replicated to a majority; no leader can be elected unless its log contains all committed entries. |

### 4.2 Leader Election – Step‑by‑Step  

Assume five nodes: \(N_1\) (current leader), \(N_2,N_3,N_4,N_5\). When the leader fails, followers detect missing heartbeats and start election:

```
Follower N2 : timeout → becomes Candidate
Candidate N2 : increment term=5, vote for self,
              send RequestVote(term=5, lastLogIndex=10) to all peers.
```

Each peer replies with `RequestVoteResponse(term=5, granted=true)` if:
- The candidate’s log is at least as up‑to‑date (Raft’s “up‑to‑date” rule), and
- It has not already voted for another candidate in this term.

If \(N_2\) receives votes from a majority (including itself) it becomes leader, starts sending heartbeats (`AppendEntries` with no entries) to indicate leadership.

### 4.3 Log Replication Example  

Consider a client command `SET x = 42`. The flow:

1. **Client → Leader N2**: `PUT key="x", value=42`.
2. **Leader N2** appends an entry `{term:5, index:11, cmd: ("set","x",42)}` to its log.
3. **Leader N2** sends `AppendEntries(term=5, prevLogIndex=10, entries=[…])` to followers.
4. Followers verify `prevLogTerm` matches their own and append the entry.
5. Followers reply with success; once leader sees acknowledgements from a majority (including itself), it *commits* the entry and notifies clients:

```
Leader N2 → Client : "OK"
```

If a follower crashes or lags, the leader retries until the log catches up. If a follower is far behind, Raft’s consistency check (`prevLogIndex`/`prevLogTerm`) forces it to discard conflicting entries – guaranteeing linearizable semantics.

### 4.4 Safety & Correctness  

Raft guarantees *safety* (no two nodes will commit different values for the same log index) via:

- **Election restriction**: A candidate can become leader only if its log contains all committed entries.
- **Commitment rule**: An entry is considered committed only after it has been replicated to a majority and the leader knows that preceding entries are also committed.

Thus, once an entry is committed, no later leader can cause a divergence.

### 4.5 Why Raft over Paxos?  

- Leader → single source of truth reduces round‑trip complexity.
- Explicit state machine transitions (follower ↔ candidate ↔ leader) map nicely to code.
- Clear log matching simplifies debugging and testing.

In practice, **etcd**, **Consul**, **TiKV**, and ** CockroachDB** use Raft for replication.

## 5. Byzantine Fault Tolerance (BFT)

When nodes can behave arbitrarily—deliberately lying or colluding—the protocols above break: a single malicious node could propose two different values to two subsets of acceptors, splitting the quorum and violating safety. BFT protocols add cryptographic validation, multiple rounds, and quorum sizes that account for up to *f* Byzantine participants while still requiring at least \(2f+1\) correct nodes.

### 5.1 The Problem Statement  

The classic formulation is the **Byzantine Generals Problem** (Lamport et al., 1982). In a replicated state‑machine setting, we want:

- **Agreement**: All non‑faulty replicas decide on the same value.
- **Validity**: If all non‑faulty propose *v*, then every non‑faulty replica decides *v*.
- **Termination**: Every non‑faulty replica eventually reaches a decision (liveness).

### 5.2 Practical Byzantine Fault Tolerance (PBFT)

PBFT, introduced by Castro and Liskov (1999), is the most widely used BFT consensus for state‑machine replication (e.g., in Hyperledger Fabric, Tendermint). The algorithm works in **views**, each managed by a *primary* (the view’s leader). It proceeds through three phases:

| Phase | Description |
|-------|--------------|
| **Pre‑prepare** | Primary assigns a sequence number *n* to the request and multicasts `Pre-prepare(view, n, digest(m))` to all replicas. |
| **Prepare** | Replicas that receive a valid pre‑prepare multicast `Prepare(view, n, digest(m), d)` (where *d* is their own digest) to all others. A replica collects a quorum of Prepare messages that match the same view and sequence number; this forms a **prepared certificate**. |
| **Commit** | After obtaining a prepared certificate, each replica multicasts `Commit(view, n, d)` to all others. Collecting a matching quorum of Commits gives a **committed certificate**, at which point the request is executed and a reply sent to the client. |

The key invariant: if any correct replica has a prepared certificate for *(v,n)* then no other correct replica can prepare a different value for that sequence number – guaranteeing safety despite up to *f* Byzantine nodes.

#### 5.2.1 Example Walkthrough  

Let’s simulate a small cluster of four nodes: \(R_0\) (primary), \(R_1,R_2,R_3\). Suppose we have one faulty node, \(R_2\), that behaves arbitrarily.

**Client C** sends `REQUEST <op>` to primary \(R_0\):

```
C → R0 : REQUEST {op: "SET x=5"}
```

*Pre‑prepare*: Primary assigns sequence number 7 and broadcasts:

```
R0 → all replicas : PRE-PREPARE(view=1, seq=7, digest=H(op))
```

**Replicas**:

- \(R_1\) receives it, checks the view and sequence number. If valid, stores it and multicasts `PREPARE`:
  
  ```
  R1 → all : PREPARE(view=1, seq=7, digest=H(op), replica=R1)
  ```

- \(R_2\) (faulty) may either ignore, send a wrong digest, or duplicate. Suppose it sends an altered message to \(R_3\).

- Each correct replica collects **2f** (= 2*1 = 2) matching Prepare messages plus its own, forming the prepared certificate.

Assume after processing:

```
R0 (primary) : has 2 PREPAREs → prepared
R1            : has 2 PREPAREs → prepared
R3 (honest)   : receives correct Prepare from R0,R1 and a wrong one from R2; discards the mismatched one, still obtains 2 matching PREPARE messages.
```

Now each replica multicasts a `COMMIT`:

```
Ri → all : COMMIT(view=1, seq=7, digest=H(op))
```

Again, any two matching commit messages (plus own) form a committed certificate. The faulty node cannot produce another set that matches but contains a different operation because it would need at least one honest replica to sign an inconsistent view.

Finally each replica executes the operation against its state machine:

```
execute(op)
reply = (result)
```

and returns result to client; client waits for *f+1* matching replies (in our example 2) – guaranteeing that at least one reply came from a correct node.

#### 5.3 View Change  

If the primary fails or behaves improperly, replicas initiate a **view change**: they exchange messages to agree on a new view number and discard any uncommitted requests; the new primary is selected by a deterministic function of the view number (often `view % N`). This ensures liveness – the system can keep making progress as long as at most *f* nodes are faulty.

### 5.4 Variants  

- **Tendermint** – a BFT consensus used in Cosmos, employing a rotating proposer and a two‑phase "prevote" + "precommit".
- **HotStuff** – introduces a “pipeline” of phases enabling linear message size for view changes; later adopted by Facebook’s Libra (now Diem) blockchain.
- **SBFT** – scales BFT to hundreds of nodes using a collector pattern to reduce O(N²) communication.

### 5.5 Performance Characteristics  

| Algorithm | Typical Latency | Communication Pattern | Fault Tolerance |
|-----------|-----------------|----------------------|-----------------|
| PBFT (4 nodes, f=1) | ≈ 3–5 round trips | O(N²) messages per request | Up to f = (N‑1)/3 |
| Tendermint | 2 rounds + block time | O(N) broadcast | Same f bound |
| HotStuff | “Pipeline” → 2 round trips amortized | O(N) with leader signature aggregation | Same |

Because BFT requires additional cryptographic signatures and larger quorums (≥ 3f+1), latency can be an order of magnitude higher than CFT solutions; however, it provides the strongest guarantee in hostile environments.

## 6. Comparison & Practical Guidance  

| Dimension | Paxos | Raft | PBFT / BFT |
|-----------|-------|------|------------|
| Failure model | Crash (CFT) | Crash (CFT) | Byzantine |
| Complexity | High – two‑phase, many corner cases | Lower – leader + log replication | Moderate – three phases plus view changes |
| Typical latency | 2–3 RTT (stable leader) | 1–2 RTT for committed entries | ≥ 3 RTT (pre‑prepare → prepare → commit) |
| Throughput | High with Multi‑Paxos pipeline | Similar to Raft but limited by single leader | Lower, due to extra signatures and quorum size |
| Implementation effort | Hard; many variations exist | Easy – reference implementation < 2K LOC | Moderate; cryptographic and state machine integration required |
| Use cases | ZooKeeper (Zab), some NoSQL clusters | etcd, Consul, TiKV, CockroachDB | Hyperledger Fabric, Tendermint, blockchain platforms |

### When to Choose What  

- **Crash‑fault only**: If you control the network and trust all participants (private cloud, internal services), use Raft for simplicity or Multi‑Paxos if you already have a robust library.  
- **Higher reliability**: etcd and Consul are examples of production CFT systems; they tolerate up to *N/2 – 1* node crashes while staying available.
- **Hostile environment**: Use BFT when the set of nodes can be partially untrusted (e.g., consortium blockchain, cross‑organizational services). Ensure you have at least \(3f+1\) nodes and a stable network; otherwise liveness may degrade.

## 7. Real-World Systems  

### 7.1 etcd – Raft in Practice  
The *etcd* project implements a production‑grade Raft library, exposing a `KV` store on top of the replicated log. The cluster elects a leader via heartbeats; clients direct all writes to the leader. Under failures, leadership may shift, and etcd's **Lease** abstraction ensures linearizable reads from followers (via read index). This design powers Kubernetes as the backing store for cluster metadata.

### 7.2 Zookeeper – Zab / Paxos  
ZooKeeper’s `zab` protocol is effectively a variant of Multi‑Paxos with a designated leader. It offers strong consistency and is used for configuration management, leader election, and distributed locks in Hadoop, Kafka, and many other Apache projects.

### 7.3 Hyperledger Fabric – PBFT  
Fabric uses a modular consensus plug-in; the default is PBFT. Clients submit transactions to peers; the ordering service (PBFT) groups them into blocks, executes the chaincode, and commits new blocks after receiving a **commit certificate**. This approach yields Byzantine‑tolerant ledger replication.

### 7.4 Tendermint – BFT for Blockchains  
Tendermint is a round‑based BFT algorithm used by many public blockchains (e.g., Cosmos). Each block production round consists of `propose` → `prevote` → `precommit`. A block is committed when > 2/3 of validators have prevoted and precommitted the same block, achieving finality without forks.

## 8. Summary  

- **Paxos** provides a mathematically proven safe solution to consensus under crash failures but is notoriously hard to implement; many systems use Multi‑Paxos to amortize overhead.
- **Raft** decomposes Paxos into intuitive components—leader election and log replication—so developers can reason about correctness more easily while preserving the same safety guarantees as Paxos in a CFT environment. Its simplicity has made it the de‑facto standard for many production systems (etcd, Consul, etc.).
- **Byzantine Fault Tolerant** protocols such as PBFT address malicious or arbitrary misbehavior by requiring larger quorums and multiple rounds of signed messages; they guarantee safety even with up to one-third Byzantine participants.

Choosing a consensus algorithm hinges on the fault model you expect, the performance budget, operational complexity, and how much trust you place in your node operators. In controlled environments where only crashes matter, Raft is usually sufficient. When you need to tolerate arbitrary failures—particularly in decentralized or multi‑tenant contexts—you must move to BFT solutions like PBFT, Tendermint, or HotStuff.

Understanding these tradeoffs arms architects with the knowledge needed to build resilient, consistent distributed systems that can survive both accidental failure and deliberate misbehavior.
